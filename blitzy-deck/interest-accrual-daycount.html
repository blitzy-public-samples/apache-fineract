The Canonical Formula

accrued_interest = principal × annual_rate × day_count_fraction
  • The amount is rounded HALF_UP to currency precision (2 dp), reusing Fineract's Money / MoneyHelper semantics.
  • Zero-day safety: when start == end, the fraction = 0 and accrued = 0.00 (no divide-by-zero).

The Three Conventions

Actual/360

Actual elapsed days over a 360-day year.

fraction = actual days ÷ 360

Actual/365 (Fixed)

Actual elapsed days over a fixed 365-day denominator.

fraction = actual days ÷ 365

30/360 US

30-day months, 360-day year, with US month-end adjustments (next slide).

fraction = 30/360 days ÷ 360
These are the only three conventions in scope.

30/360 US — the month-end rules

days = 360·(Y2−Y1) + 30·(M2−M1) + (D2−D1)   then ÷ 360
  • Adjustment order: if D1 = 31 then D1 → 30; then if D2 = 31 and D1 ∈ {30, 31} then D2 → 30.
  • Key behavior: end-day 31 is not adjusted when the start day is not 30/31 — which is why scenario S4 (2025-01-01 → 2025-03-31) yields 90 days.

Design Principles

Additive & backward-compatible

One optional JSON field accrualDayCountConvention; a nullable DB column; NULL = today's behavior byte-for-byte (no silent change).

Minimal change & isolation

New logic in dedicated files; exactly one production method behaviorally changed; every edit to an existing file tagged // [Day-Count Convention feature].

Reuse, don't reinvent

Reuse Money/MoneyHelper rounding, DateUtils, the existing accrual engine and GL engine. No parallel engine, no new numeric approach, no new third-party dependency.

Safety

Null-guarded everywhere; the zero-day boundary accrues 0.00 with no divide-by-zero.

The Single Change Point + GL Reuse

BEFORE  Actual days only

interestPerDay = interest ÷ totalDays
portion = interestPerDay × daysToAccrue

Period interest was apportioned by actual elapsed days only — no convention selectability.

AFTER  Convention-aware

When a convention is set, the day-count fraction comes from the calculator; when it is null, the existing actual-day path runs unchanged.

The corrected Money amount then flows through the existing journal-entry mechanics — the General Ledger, chart of accounts, and double-entry engine are untouched.
